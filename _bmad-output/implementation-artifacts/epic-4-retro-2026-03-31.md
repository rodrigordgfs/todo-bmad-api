# Retrospectiva do Epic 4: Descoberta e Consistencia de Contrato

Data: 2026-03-31
Epic: 4
Status: concluido

## Participantes

- Rodrigo (Project Lead)
- Codex (Dev / Review / Facilitação)

## Resumo do Epic

O Epic 4 consolidou a camada de descoberta e acabamento do MVP: a API passou a oferecer busca textual case-insensitive, composição previsível entre busca e filtro por estado e um contrato final mais explícito para erro, sucesso e documentação Swagger.

**Stories concluidas**

- `4.1` Busca textual case-insensitive em campos relevantes
- `4.2` Combinar busca com filtro de estado
- `4.3` Refinar contrato de resposta para feedback e recuperacao de contexto

## O Que Funcionou Bem

- A disciplina de contrato HTTP continuou forte mesmo com a expansão da listagem: `search` entrou sem reabrir parâmetros fora de escopo.
- A composição entre `status`, `search` e ordenação ficou explícita no fluxo da feature e não dependente de comportamento implícito.
- O padrão arquitetural se manteve estável: controller enxuto, service centralizando semântica de domínio e repository isolando Prisma.
- O Swagger deixou de ser apenas um extra e passou a refletir melhor o comportamento real do MVP, inclusive no contrato de erro.

## Principais Desafios

- Expandir a listagem sem relaxar a validação estrita de query params exigiu decisões cuidadosas de contrato.
- A busca textual precisou ser definida de forma útil para produto sem gerar ambiguidade para `search` vazio, whitespace ou ausência completa.
- A composição entre busca, filtro e ordenação exigiu clareza para evitar regressões silenciosas em comportamento que já estava estável desde o Epic 3.
- Durante a story `4.3`, houve um desvio estrutural no workspace com ausência de arquivos base do app, o que exigiu restauração para recompor build e testes reais.

## Insights e Aprendizados

1. Busca textual nao e detalhe tecnico.
Assim que `search` entra na API, ele passa a fazer parte do contrato do produto e precisa de semântica explícita para casos comuns e de borda.

2. Composicao precisa ser declarada, nao presumida.
Definir claramente a ordem `status -> search -> ordenacao` evitou leituras divergentes entre implementação, review e expectativa de uso.

3. Contrato honesto reduz retrabalho.
Continuar rejeitando parâmetros fora do escopo ajudou a impedir integrações enganosas e manteve a API previsível.

4. Documentacao madura acompanha comportamento maduro.
Quando a feature `tasks` atingiu um nível mais estável, Swagger passou a ser parte do produto entregue, não só apoio técnico.

## Follow-through da Retrospectiva Anterior

A retrospectiva do Epic 3 trouxe quatro focos principais. Neste epic:

- ✅ Introduzir busca textual sem relaxar a validação estrita de query params.
  Evidência: `search` foi incorporado mantendo schema estrito e rejeição de parâmetros fora do escopo.
- ✅ Combinar busca com filtro por estado de forma previsível.
  Evidência: a composição foi formalizada e testada como `status -> search -> ordenacao`.
- ✅ Preservar a ordenação útil da lista ao adicionar novas queries.
  Evidência: a ordenação determinística continuou valendo após busca e filtro combinado.
- ✅ Reavaliar o papel do service na semântica da listagem.
  Evidência: a lógica continuou centralizada no service com bom controle, mas o risco de futura migração para a persistência ficou mais evidente.

## Descobertas Significativas

- A feature `tasks` agora oferece um contrato de listagem mais completo:
  - filtro por estado
  - busca textual case-insensitive
  - composição previsível entre filtros
  - ordenação determinística estável
- O contrato de erro e a documentação OpenAPI ficaram mais próximos do comportamento real da aplicação.
- O review continuou útil para capturar ambiguidades de semântica antes que elas virassem comportamento acidental.

## Dividas e Riscos Residuais

- A lógica de listagem ainda está bastante centralizada no service; se entrarem paginação, volume maior ou busca mais sofisticada, parte dessa semântica provavelmente precisará descer para a persistência.
- Os testes e2e continuam valiosos, mas seguem dependentes de PostgreSQL disponível no ambiente.
- A documentação Swagger agora está melhor alinhada, mas futuras expansões de contrato podem pedir decorators reutilizáveis para reduzir repetição.

## Preparacao para o Proximo Ciclo

### Preview

O MVP funcional da API ficou consolidado ao final do Epic 4.

O próximo ciclo natural pode seguir uma de duas direções:

- endurecimento operacional da API, com foco em paginação, performance, observabilidade e robustez
- expansão funcional do domínio, caso existam novas capacidades de produto além do MVP atual

### Dependencias Criticas

- decidir se a próxima evolução prioriza escala técnica ou novas features
- manter a honestidade do contrato HTTP se novos query params surgirem
- reavaliar quando a lógica de listagem deve migrar parcialmente para a camada de persistência
- preservar o alinhamento entre comportamento real, testes e Swagger

## Acoes Recomendadas

- Rodrigo: revisar o estado final do sprint e decidir se o próximo passo é fechamento do MVP ou abertura de um novo ciclo de planejamento.
- Rodrigo: tratar paginação e performance como candidatos naturais para o próximo conjunto de histórias, se a API for crescer em volume.
- Codex: continuar protegendo o contrato HTTP com validação estrita e reviews focados em aceitação silenciosa de parâmetros.
- Codex: revisitar a distribuição de responsabilidade entre service e repository quando surgirem requisitos de escala ou paginação.

## Readiness do Proximo Ciclo

O projeto está pronto para encerrar este ciclo do MVP e decidir o próximo foco.

Motivos:

- o CRUD de `tasks` está completo
- estado, filtro, busca e ordenação estão cobertos
- o contrato de erro e a documentação Swagger foram consolidados
- os principais riscos residuais estão identificados e têm direção de mitigação clara

## Encerramento

O Epic 4 foi o fechamento de qualidade do MVP: ele não apenas adicionou busca e composição de filtros, mas também amadureceu a API como produto, com contrato mais claro, documentação mais confiável e comportamento mais previsível. O principal aprendizado foi que consistência de listagem e consistência de contrato andam juntas; quando uma falha, a outra inevitavelmente degrada junto.
