# Retrospectiva do Epic 1: Fundacao Operacional da API de Tarefas

Data: 2026-03-30
Epic: 1
Status: concluido

## Participantes

- Rodrigo (Project Lead)

## Resumo do Epic

O Epic 1 entregou a fundacao operacional da API com NestJS, Prisma, PostgreSQL, bootstrap versionado em `/api/v1`, Swagger, CORS configuravel e contrato global de erro com validacao via `zod`.

**Stories concluidas**

- `1.1` Inicializar projeto NestJS com base tipada e estrutura modular
- `1.2` Configurar Prisma e conexao PostgreSQL com migrations iniciais
- `1.3` Padronizar bootstrap HTTP, CORS, versionamento e Swagger
- `1.4` Implementar contrato global de erro e validacao de entrada

## O Que Funcionou Bem

- A sequencia incremental das stories funcionou bem: primeiro fundacao, depois persistencia, depois bootstrap HTTP e por fim contrato transversal.
- O fluxo `DS -> CR -> ajuste -> CR` evitou que detalhes importantes escapassem, especialmente em configuracao, testes e superficies expostas do app.
- O projeto terminou o epic com uma base limpa para evoluir: arquitetura modular, bootstrap centralizado e ambiente de testes reproduzivel.
- As validacoes recorrentes com `lint`, `build`, testes unitarios e e2e ajudaram a detectar regressao cedo.

## Principais Desafios

- Parte da fundacao exigiu diferenciar claramente o que era setup estrutural e o que seria implementacao prematura de dominio.
- Houve risco recorrente de dependencias implícitas de ambiente local, especialmente em `DATABASE_URL` e no setup de testes.
- Algumas provas tecnicas inicialmente resolveram o teste, mas criavam risco no runtime real, como o modulo `foundation` exposto fora do ambiente de teste.
- Story de bootstrap mostrou que versionamento e Swagger parecem simples, mas escondem armadilhas pequenas com impacto real, como rota minima sem versao e path relativo do JSON OpenAPI.

## Insights e Aprendizados

1. Fundacao boa precisa de guardrails explicitos.
Sem isso, a implementacao tende a resolver o sintoma do teste e nao o comportamento esperado do sistema.

2. Configuracao reprodutivel precisa ser tratada como parte da feature.
Quando o teste depende de `.env` local nao versionado, o ganho aparente vira fragilidade para CI e clone limpo.

3. Code review agregou valor real neste epic.
Os principais ajustes surgiram em review, nao na primeira passada de implementacao: fallback de `DATABASE_URL`, estrategia de setup de teste, `jsonDocumentUrl` do Swagger e isolamento do modulo tecnico de prova.

4. Provas tecnicas devem existir, mas fora do runtime produtivo.
Rotas ou modulos de apoio para e2e sao uteis, desde que fiquem confinados ao ambiente de teste.

## Descobertas Significativas

- O bootstrap compartilhado em `src/config/app.config.ts` virou um ponto forte de consistencia entre runtime e e2e.
- A combinacao de `zod` + filtro global + contrato estavel de erro deixou a API preparada para o frontend antes mesmo do CRUD de `Task`.
- O projeto ficou com uma base favoravel para o Epic 2: infraestrutura pronta, contrato HTTP claro e testabilidade melhor do que no inicio do sprint.

## Dividas e Riscos Residuais

- O `.env.test` ainda aponta para um PostgreSQL local. Quando o Epic 2 passar a executar queries reais, sera necessario garantir banco disponivel no fluxo de teste ou isolar melhor esse setup.
- O pipeline de migrations foi preparado, mas a primeira migration real ainda depende da `2.1`.
- O contrato de erro esta pronto, mas ainda sera importante mantê-lo consistente quando entrarem erros de dominio e persistencia reais.
